# Отчет о тестировании

## 1. Цель тестирования

Цель тестирования — проверить корректность запуска проекта, работу основных пользовательских сценариев приложения для заказа еды, а также выявить ошибки интерфейса, навигации и бизнес-логики.

## 2. Среда тестирования

Операционная система: macOS  
Браузер: Google Chrome / Safari  
Способ запуска: Docker Compose  
Frontend: Next.js  
Backend: FastAPI

## 5. Найденные дефекты

### Баг 1. Нельзя полностью очистить поле номера телефона

Экран: экран авторизации / ввода номера телефона.

Шаги воспроизведения:

1. Открыть стартовую страницу приложения.
2. Начать ввод номера телефона.
3. Ввести номер некорректно, например начать с 8 вместо 7.
4. Попытаться полностью удалить введённый номер и ввести его заново.

Ожидаемый результат:

Пользователь может полностью очистить поле номера телефона и ввести номер заново.

Фактический результат:

Поле не очищается полностью. После удаления остаётся значение вида «+7 (7», из-за чего пользователь не может нормально заново ввести номер.

Критичность: средняя.

Статус: найден, передан команде.

### Баг 2. Ссылка «условиями использования» не кликабельна

Экран: экран авторизации / ввода номера телефона.

Шаги воспроизведения:

1. Открыть стартовую страницу приложения.
2. Найти текст «Нажимая кнопку, вы соглашаетесь с условиями использования».
3. Нажать на выделенный текст «условиями использования».

Ожидаемый результат:

Открывается страница, модальное окно или документ с условиями использования.

Фактический результат:

Текст визуально выделен как ссылка, но не является кликабельным и никуда не ведёт.

Критичность: низкая.

Статус: найден, передан команде.

---

### Баг 3. На экране меню кнопка «−» удаляет все блюда одной позиции

Экран: меню / список блюд.

Шаги воспроизведения:

1. Открыть экран меню.
2. Добавить одно блюдо несколько раз с помощью кнопки «+».
3. Нажать кнопку «−» у этой позиции на экране меню.

Ожидаемый результат:

Количество выбранного блюда уменьшается на 1.

Фактический результат:

Позиция полностью удаляется из выбранных блюд, даже если количество было больше 1.

Дополнение:

На экране корзины эта логика работает корректно: при нажатии «−» количество уменьшается на 1.

Критичность: средняя.

Статус: найден, передан команде.

---

### Баг 4. В панели категорий есть неработающие кликабельные категории

Экран: меню / верхняя панель категорий блюд.

Шаги воспроизведения:

1. Открыть экран меню.
2. Нажать на категории в верхней панели, например «Супы», «Салаты», «На огне», «Гарниры», «Соусы», «Десерты», «Напитки».
3. Проверить переход или прокрутку к соответствующему разделу.

Ожидаемый результат:

Если категория отображается как кликабельная, то при нажатии пользователь должен попадать к соответствующему разделу меню. Если блюд в категории нет, категория не должна отображаться или должна быть визуально недоступной.

Фактический результат:

Некоторые категории визуально выглядят кликабельными, но не ведут к соответствующему разделу.

Критичность: средняя.

Статус: найден, передан команде.

### Баг 5. Стрелки прокрутки категорий активны на границах списка

Экран: меню / верхняя панель категорий блюд.

Шаги воспроизведения:

1. Открыть экран меню.
2. Пролистать категории до крайнего левого или крайнего правого положения.
3. Нажать на жёлтую стрелку прокрутки дальше в сторону конца списка.

Ожидаемый результат:

Если пользователь уже находится в начале или конце списка категорий, соответствующая стрелка должна быть скрыта или неактивна.

Фактический результат:

Стрелка остаётся активной и выглядит кликабельной, хотя дальше список не прокручивается.

Критичность: низкая.

Статус: найден, передан команде.

### Баг 6. Кнопка адреса открывает неверную локацию в Google Maps

Экран: меню / верхняя панель приложения.

Шаги воспроизведения:

1. Открыть экран меню.
2. Нажать на адрес в верхней панели: «Пр. Добролюбова 7/2».
3. Дождаться открытия Google Maps.

Ожидаемый результат:

Открывается точная локация ресторана по адресу «Пр. Добролюбова 7/2».

Фактический результат:

Google Maps открывает не ту локацию. Точка находится рядом с нужным местом, но не совпадает с фактическим адресом.

Критичность: средняя.

Статус: найден, передан команде.

### Баг 7. Нет страницы со списком заказов и невозможно вернуться к активному заказу

Экран: страница статуса заказа / меню / профиль пользователя.

Шаги воспроизведения:

1. Оформить заказ.
2. Попасть на страницу статуса заказа.
3. Нажать «Вернуться в меню».
4. Попытаться снова открыть страницу текущего заказа или посмотреть список своих заказов.

Ожидаемый результат:

У пользователя должна быть возможность открыть список своих заказов и перейти к активному заказу, чтобы проверить его номер, состав и текущий статус.

Фактический результат:

После возврата в меню пользователь не может снова открыть страницу активного заказа. Если пользователь забыл номер заказа, он не может восстановить информацию о заказе внутри приложения.

Критичность: высокая.

Статус: найден, передан команде.

Экран: экран ввода SMS-кода.

Шаги воспроизведения:

1. Ввести корректный номер телефона.
2. Перейти на экран ввода кода.
3. Ввести любой произвольный код.

Приложение должно проверять код подтверждения. При неверном коде пользователь должен получить сообщение об ошибке.

Фактический результат:

Приложение принимает любой введённый код и пропускает пользователя дальше.

Критичность: высокая.

Статус: найден, передан команде.

Фактический результат:

После возврата номер телефона пропадает, поле становится пустым.

Критичность: низкая.

Статус: найден, передан команде.

## 7. Вывод

Базовый запуск проекта через Docker выполнен успешно: frontend и backend доступны, Swagger открывается, health-check backend работает корректно.

Во время ручного тестирования пользовательских сценариев было найдено 9 дефектов. Наиболее критичные проблемы связаны с отсутствием проверки SMS-кода и невозможностью вернуться к активному заказу после перехода обратно в меню.

Также обнаружены ошибки в логике изменения количества блюд, навигации по категориям, работе поля номера телефона, ссылок интерфейса и адреса ресторана.

Рекомендуется исправить найденные дефекты перед итоговой демонстрацией проекта.
